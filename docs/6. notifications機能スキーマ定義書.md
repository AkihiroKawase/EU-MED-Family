## 6.1. notifications コレクションの運用ルール
- notificationsコレクションは、特定のアクション（いいね、コメント、返信など）が発生した際に、関連するユーザーへ通知を送るための記録を保持します。
- 1つのドキュメントが、1ユーザーへの1件の通知に対応します。
- このドキュメントは、Cloud Functionsなどがサーバーサイドで（例: likes や comments の作成をトリガーとして）作成します。クライアント（アプリ）が直接書き込むことはありません。
- 通知には「種類（type）」を持たせ、いいね・コメント・返信などを区別します。
- 通知には「既読（isRead）」フラグを持たせ、ユーザーが通知一覧を開いた際に true に更新できるようにします。

## 6.2. フィールド: notifications に必要なフィールド
- userId: (String) 通知の受信者のID（例: 投稿の作者）
- type: (String) 通知の種類。列挙型（NEW_LIKE, NEW_COMMENT, NEW_REPLY など）
- triggerUserId: (String) 通知を発生させたユーザーのID（例: いいねした人）
- triggerUserName: (String) 通知を発生させたユーザーの名前（非正規化。通知一覧での表示を高速化するため）
- postId: (String) 関連する投稿のID（通知タップ時の遷移先）
- commentId: (String, Optional) 関連するコメントのID（NEW_REPLY の場合など）
- message: (String) 通知一覧に表示する本文。（例: 「XXさんがあなたの投稿に「いいね」しました」）
- isRead: (Boolean) 既読フラグ。デフォルトは false。
- createdAt: (Timestamp) 通知が作成された日時。

##6.3. インデックス（推奨）
- ユーザーが「自分の通知一覧」を「新しい順」で取得するユースケースが主となるため、以下の複合インデックスを強く推奨します。
- (userId, createdAt desc)
